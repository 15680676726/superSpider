# UNIFIED_ACCEPTANCE_STANDARD.md

本文件是 `CoPaw` 仓库的统一验收规范。

它不是某一轮专项验收文档，也不是某个版本的临时补充，而是以后所有功能、修复、重构、删旧、读面调整、主链收口都要共同遵守的默认验收口径。

目标只有一个：

- 不再把“单测通过”误写成“真实可用”
- 不再把“接口 200”误写成“主链闭环”
- 不再把“代码存在”误写成“系统已经真的会跑”

---

## 1. 核心原则

### 1.1 没有新证据，不得宣称完成

任何“已完成”“已通过”“可上线”“闭环了”的表述，都必须对应本轮真实跑出来的证据。

允许作为证据的内容至少包括：

- 本轮命令与结果
- 本轮 live 跑通记录
- 本轮产物路径
- 本轮失败归因与剩余边界

### 1.2 `200`、不报错、单点通过，都不等于真实运行通过

以下情况都不能单独视为“真实可用”：

- 接口返回 `200`
- 单个 service 单测通过
- TestClient 通过
- 某个页面静态渲染通过
- 某条 happy path 人工点通一次

这些只能证明局部没立刻炸，不能证明整条链真的闭环。

### 1.3 功能存在，不等于对人正确

系统里“有这个对象”“有这条接口”“有这张卡片”“有这段摘要”，都不代表它对用户读面是正确的。

必须继续核对：

- 写进去的是真对象，还是临时态
- 下游读的是同一份真相，还是旧字段
- 前端显示的是最新对象，还是旧缓存 / 旧 fallback / 第一条记录
- 刷新、重启、并发后还对不对

### 1.4 必须分层表达完成度

以后仓库里所有完成声明，必须显式区分下面四层：

- `L1`：单元验证
- `L2`：集成 / 合同验证
- `L3`：真实运行 smoke
- `L4`：长链 soak

不允许再把不同层级混写成一句“测试通过”。

---

## 2. 四层验收模型

### 2.1 `L1` 单元验证

验证局部规则、函数、对象、排序、判定、格式化、映射是否正确。

典型内容：

- 纯函数
- repository 单对象读写
- 判定逻辑
- 排序与优先级
- 文本整理 / 摘要拼装
- schema / migration 单点校验

`L1` 解决的是“这段代码本身有没有明显写错”，不解决“系统真实运行时会不会串”。

### 2.2 `L2` 集成 / 合同验证

验证链路之间的数据合同和主链接线是否一致。

每条关键链至少核对 4 件事：

1. 输入对象是什么
2. 落库对象是什么
3. 下一层读的字段是什么
4. 前端或查询读面最终读的又是什么

重点不是“调到了”，而是：

- 写的是不是正式对象
- 读的是不是同一份真相
- 有没有旧字段 / 旧接口 / 旧 fallback 混入
- 有没有临时态覆盖正式态

### 2.3 `L3` 真实运行 smoke

验证系统在真实运行条件下能不能闭环，不是只在测试桩里通过。

至少应尽量包含下面中的若干项：

- 真起服务
- 真走 HTTP / 正式前门
- 真创建对象
- 真执行一次主链
- 真刷新前端读面或查询读面
- 真做一次恢复 / reload / restart 后复核

`L3` 主要回答：

- 真实入口通不通
- 主链有没有断点
- 用户实际看到的内容对不对

### 2.4 `L4` 长链 soak

验证系统不是“跑一次看起来没事”，而是多轮运行后仍然不脏、不串、不漂。

至少应考虑：

- 连续多轮同一条链
- 中途插入干扰与冲突
- 重启恢复
- 定时唤醒
- 多 agent 并发
- 多 scope 混跑
- 前端停留旧页签后的脏读

`L4` 主要回答：

- 系统会不会越跑越脏
- 状态会不会慢慢串
- 旧缓存 / 旧 fallback / 异步时序会不会把真相冲掉

---

## 3. 场景优先，不按零散功能点验收

统一验收默认优先验证“真实闭环场景”，不是只列一堆零散功能点。

推荐先列最关键的真实流程，例如：

- 用户下达目标 -> 主脑形成正式对象 -> 进入执行链
- 主脑派工 -> agent 接单 -> 执行 -> 产出证据 -> 回写主脑
- 前端读取运行真相 -> 刷新后仍显示同一对象
- 定时任务触发 -> 系统苏醒 -> 跑完一轮 -> 状态刷新
- 重启后恢复 -> 仍读到正确 runtime / memory / focus

如果某项“功能”离不开上下游，验收就不能只盯它自己。

---

## 4. 每条关键流程都要做断点表

每条关键流程至少检查这 8 个断点：

1. 入口事件
2. 正式对象是否创建
3. 状态是否持久化
4. 执行是否真的发生
5. 证据是否生成
6. 写回是否完成
7. 前端 / 查询读面是否一致
8. 刷新、恢复、重进后是否仍一致

任何一步只停留在“返回了一段看起来像成功的话”，都不算通过。

---

## 5. 失败归因必须写到根类

以后验收失败，不允许只写“功能异常”“偶发问题”“显示不对”。

至少归到以下根类之一：

- 状态没落正式对象
- 字段名 / 数据合同不一致
- 临时态覆盖正式态
- 前端读了旧接口 / 旧字段 / 旧 fallback
- 执行发生了但没写证据
- 证据写了但没回写主链
- 环境 / 能力存在但没真正挂载
- 恢复 / 刷新后读错实例或线程
- 异步时序问题
- 缓存脏读

只有归因到根类，后续修复才不会一直补表象。

---

## 6. 最低验收门槛矩阵

### 6.1 文档-only 改动

最低要求：

- 文档一致性检查
- 相关总规约 / 状态文档同步

### 6.2 纯局部纯函数 / 纯格式化 / 纯静态映射

最低要求：

- `L1`

### 6.3 repository / service / API / schema / migration / 数据合同

最低要求：

- `L1 + L2`

### 6.4 用户可见读面、排序、摘要、卡片、前端展示

最低要求：

- `L1 + L2 + L3`

原因很简单：这类改动最容易出现“后端是对的，但用户看到的是旧的”。

### 6.5 以下高风险主链

包括但不限于：

- memory
- sleep
- graph
- recall
- ranking
- scheduler / cron
- recovery / restart
- runtime center
- 多 agent 并发
- host / environment / capability activation
- 主脑 / assignment / report / writeback 主链

最低要求：

- `L1 + L2 + L3 + L4`

这类链路只做短链验证基本一定不够。

## 7. 完成声明规则

以后任何完成声明必须写清楚“通过了哪一层，没通过哪一层”。

允许的写法：

- 已通过 `L1 + L2`，`L3` 未跑
- 已通过 live smoke，但尚未完成 long soak
- 当前只证明代码基线通过，不等于默认真实外部可用

不允许的写法：

- 测试通过，所以已经可用
- 已完成
- 已闭环
- 没问题了

如果因为环境限制没法跑 `L3` 或 `L4`，必须明确写：

- 没跑哪一层
- 为什么没跑
- 当前结论只能到哪一层

### 7.1 按文档 / spec / plan 逐条核对时的强制格式

凡是用户要求“按文档逐条核对是否完成”，默认只能按条目化格式汇报，不允许先给总括结论。

每一条至少要带：

- 条目名
- 状态：`已实现 / 部分实现 / 未实现 / 未验证`
- 代码文件
- 测试命令
- 测试结果
- 验收层级：`L1 / L2 / L3 / L4`
- 剩余缺口

硬规则：

- 没有代码证据，不得写 `已实现`
- 没有测试证据，不得写“已验证完成”
- 没有 `L3 / L4` 证据，不得把 `L1 / L2` 混写成“真实运行通过”
- 只要文档中仍有任一条是 `部分实现 / 未实现 / 未验证`，就不得把整份文档写成“已完成 / 已闭环 / 全部没问题”

禁止的混写方式包括：

- 把 `phase-1` / “基线已落地”写成“整套完成”
- 把“有对象 / 有接口 / 有前门”写成“功能闭环”
- 把“文档写了 / 计划列了”写成“代码已经实现”

推荐模板：

```text
文档：<path>

1. 条目：<name>
状态：已实现 / 部分实现 / 未实现 / 未验证
代码：<file refs>
测试：<commands>
结果：<outputs>
验收层级：L1 / L2 / L3 / L4
缺口：<remaining gaps>
```

## 8. `TASK_STATUS.md` 记录规则

以后 `TASK_STATUS.md` 里的每条“通过 / 已完成 / 已闭环”记录，必须尽量带齐下面信息：

- 跑了什么命令
- 结果是什么
- 属于 `L1/L2/L3/L4` 哪一层
- 是否是真实 live
- 是否是 opt-in live
- 是否做过 soak
- 当前剩余边界是什么

必须拆开写的概念：

- `default regression`
- `focused regression`
- `live smoke`
- `long soak`

不允许再把“代码基线通过”写成“真实运行已验证”。

---

## 9. 前后端一致性专项规则

前端问题最容易出在“后端真相对了，但页面还在读旧链”。

凡是用户可见页面，至少额外核对：

- 页面首读走的接口是什么
- 切页 / 刷新 / 轮询后走的接口是不是同一条
- fallback 读的是不是旧字段
- `.find(...)` / `[0]` / 默认第一条 这类旧逻辑有没有把最新对象盖掉
- 多次请求先后返回时，会不会被旧响应覆盖新响应

如果没有真浏览器级 e2e，也不能把前端真实运行风险写成“已全部消失”。

---

## 10. 记忆 / 睡眠 / 图谱专项规则

这类链路最容易出现“写进去了，但下一轮没读对”。

至少要额外验：

- 写入的是正式事实，还是只写了派生层
- sleep 后 overlay / profile / proposal 有没有对齐 active truth
- recall 读面是不是先读正式 overlay/profile，再读图谱 / evidence
- 噪音信息会不会劫持主记忆
- 标题弱、摘要弱时会不会被旧记录抢读
- 重启后记忆主读层是否仍一致
- 多轮后会不会慢慢脏

没有这些验证，就不能把“有记忆表了”写成“记忆系统已完整可用”。

---

## 11. 发布 / 合并前的最低口径

未达到该改动类型的最低验收门槛前：

- 不得写“完成”
- 不得写“闭环”
- 不得写“可以上线”

如果当前只能证明到较低层级，就老实按较低层级写。

诚实边界不是减分项，混写才是。

---

## 12. 推荐执行顺序

以后做任何重要功能，默认按这个顺序验：

1. 先补 `L1`，锁住局部规则
2. 再补 `L2`，锁住正式对象和数据合同
3. 再跑 `L3`，确认真实入口和读面闭环
4. 最后视风险补 `L4`，确认不会越跑越脏

如果一上来只跑单点 happy path，最后大概率还会回头返工。

---

## 13. 一句话总规则

以后在 `CoPaw` 里，任何“通过”“完成”“可用”的说法，都必须回答清楚：

- 通过的是哪一层验收
- 真正跑通的是哪条主链
- 证据在哪里
- 剩下哪一层还没过

答不出来，就不算真正验收完成。
