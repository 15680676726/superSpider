# CoPaw 行业记忆结构增强设计

日期：`2026-04-16`

---

## 0. 目标

这次不是再造一套记忆系统，也不是重做知识图谱。

这次要做的是把当前已经存在的：

- 正式事实层
- 知识图谱层
- `B+` 睡眠整理层

真正收成一条更完整的行业长期记忆链，让系统不只是“存得下”，而是真的能在下一轮稳定读到最重要的连续性重点。

一句话目标：

> 让睡眠层站在正式事实与知识图谱之上，按行业和阶段自动整理连续性锚点、强写入关键细节，并把治理、前台读面与真实执行一起闭环。

---

## 1. 这次要解决的真问题

当前系统已经具备：

- 正式事实真相
- 正式知识图谱与关系视图
- `industry_profile / work_context_overlay / structure_proposal`
- `Runtime Center / Knowledge` 的正式读面基础

但还存在 5 个关键不足：

1. 图谱已经有关系，但还不够直接服务“下一轮最该先读什么”
2. 睡眠层已经会做摘要，但还没有稳定的行业结构增强能力
3. 高价值长期细节还缺少更强的正式写入链
4. `proposal apply` 当前只是治理接受，不是真正重写 active 结构
5. 前台虽然能看到 memory surface，但还没有把“连续性锚点 / 强写入重点 / 行业偏好结构”压成真正易读的正式面

这次设计就是把这 5 个点一起收口，不拆成“以后再说”。

---

## 2. 总结构

这次仍然坚持 5 层，不增加第二真相源：

1. 正式事实层
   - 负责记住真实发生了什么
   - 包括 `strategy / backlog / cycle / assignment / report / evidence / knowledge chunk / work_context`

2. 图谱层
   - 负责记住对象、关系、因果、证据链
   - 它是关系网络，不直接充当“次日优先读面”

3. 睡眠整理层
   - 负责产出：
     - `IndustryMemoryProfileRecord`
     - `WorkContextMemoryOverlayRecord`
     - `MemoryStructureProposalRecord`
   - 它要从“做摘要”升级为“提炼连续性锚点”

4. 行业记忆结构层
   - 不新增第二套数据库真相
   - 只定义睡眠层该如何组织、提权、沉淀行业重点

5. 前台读面层
   - `Runtime Center` 只显示简化结果
   - `Knowledge` 显示结构化细节
   - 本轮不把正式行业记忆结构接入 `Chat`

---

## 3. 统一底座与行业增强

### 3.1 固定通用主槽位

不管什么行业，正式结构都先固定这 8 类主槽位：

- 关键对象
- 关键关系
- 硬规则 / 约束
- 已承诺事项
- 时间线 / 阶段位置
- 当前阶段焦点
- 证据缺口 / 冲突点
- 下一步顺序

这 8 类槽位是稳定骨架，不允许每天换脑子。

### 3.2 动态增强槽位

在固定骨架之上，模型可以根据：

- 当前行业
- 当前阶段
- 当前工作上下文
- 当前图谱热点

补出本轮特别重要的增强槽位。

典型示例：

- 小说：
  - 角色状态
  - 世界规则
  - 剧情承诺
  - 伏笔回收
- 漫剧：
  - 角色出场连续性
  - 分镜节奏
  - 场景承接
  - 镜头约束
- 研究：
  - 核心假设
  - 变量
  - 反例
  - 验证顺序
- 交易：
  - 当前仓位前提
  - 风险边界
  - 禁止动作
  - 触发条件

这些增强槽位不应各自新建一套行业专用表，而应作为统一结构中的增强视图和优先级信号。

### 3.3 长期偏好升格

模型动态补出的槽位，不直接升格成行业长期规则。

只有满足门槛才可升格为该行业的长期偏好槽位：

- 连续多轮稳定出现
- 连续多轮被真实读到或真实使用
- 对结果连续有帮助
- 不与正式事实和治理边界冲突

### 3.4 长期偏好降权

已经升格的长期偏好也不是永久有效。

出现以下情况应自动降权：

- 长期未被读到或未被使用
- 多轮与更新后的正式事实冲突
- 被更稳定的新槽位替代

结论：

- 不是固定死模板
- 也不是完全动态乱变
- 而是“稳定骨架 + 动态增强 + 行业偏好沉淀 + 无效降权”

## 4. 图谱如何正式喂给睡眠层

当前知识图谱已经是正式链的一部分，这次不重做图谱。

要补的是：让睡眠层真的把图谱当成连续性输入，而不是只把文本摘要当主材料。

### 4.1 睡眠层的图谱输入

睡眠层在生成 `industry_profile / work_context_overlay` 时，必须显式读取：

- 当前 scope 最活跃的关键对象
- 当前最稳定的关键关系
- 当前阶段最相关的因果链
- 当前存在冲突或缺口的关系点
- 最近被反复引用的证据链

### 4.2 图谱输入的用途

图谱不是为了让睡眠层直接抄 relation 列表。

它的用途是帮助睡眠层回答：

- 这轮最不能忘的对象是谁
- 哪几条关系最影响后续连续性
- 哪条规则 / 承诺如果忘了最容易翻车
- 下一步应沿哪条因果链继续推进

### 4.3 睡眠层的产出升级

升级后，`IndustryMemoryProfileRecord` 与 `WorkContextMemoryOverlayRecord` 不只存摘要文本，还要显式体现：

- 连续性锚点
- 当前优先关系
- 当前关键对象
- 当前冲突 / 缺口
- 当前行业偏好命中的增强槽位

---

## 5. 关键细节强写入

这条链必须单独做，不然长期连续性永远会掉。

### 5.1 强写入目标

需要被强写入的不是所有细节，而是高价值长期细节，例如：

- 角色设定
- 世界规则
- 剧情承诺
- 研究假设
- 风险边界
- 禁止动作
- 镜头承接约束
- 关键身份 / 关系变化
- 明确时间线锚点

### 5.2 判定方式

采用：

`模型主判 + 规则兜底 + 人工钉住`

#### 模型主判

模型负责识别哪些细节属于“长期连续性高风险点”。

#### 规则兜底

规则负责兜住明显不能漏掉的内容，例如：

- 明确约束
- 已承诺但未完成事项
- 禁止动作
- 阶段性硬前提
- 明确时间线节点

#### 人工钉住

在 `Knowledge` 中允许用户把某条细节标成：

- 必须长期记住
- 在某阶段结束前不要降权

### 5.3 强写入落点

强写入细节不能只躺在普通 chunk 里。

它们应稳定进入：

- `WorkContextMemoryOverlayRecord` 的高优先结构字段
- `IndustryMemoryProfileRecord` 的长期偏好摘要字段
- 必要时进入图谱节点 / 关系的高权重引用

同时必须保留来源 evidence，防止模型脱离事实胡编。

### 5.4 边界

强写入只允许：

- 提权
- 聚合
- 归并表达
- 固定进入优先读面

不允许：

- 篡改 evidence
- 篡改正式事实
- 擅自改变长期规则
- 越过治理边界直接生效高风险结论

## 6. `proposal apply` 真执行

当前 `POST /runtime-center/memory/sleep/structure-proposals/{proposal_id}/apply` 的真实语义只是：

- 把 proposal 状态改成 `accepted`

这不够。

### 6.1 正确语义

`apply` 应变成真正的结构执行器：

- 校验 proposal 是否仍然对齐当前 active truth
- 必要时重算目标结构
- materialize 新版本 active overlay/profile
- 将旧版本置为 `superseded`
- 保留来源 proposal、来源 sleep job、来源 evidence、diff

### 6.2 真执行后的读链要求

一旦 apply 成功，以下入口必须直接读到新 active 版本：

- recall
- profile
- `/runtime-center/memory/surface`
- `Runtime Center`
- `Knowledge`

不能再出现“批准了，但系统还是读旧结构”的假闭环。

## 7. 前台读面

这次前台只做简化，不追求炫技。

### 7.1 `Runtime Center`

只显示：

- 当前最该记住的连续性锚点
- 当前关键约束
- 当前关键风险 / 冲突
- 待处理结构提案
- 最近一次睡眠整理结果摘要

### 7.2 `Knowledge`

显示完整结构化细节：

- 当前行业记忆基线
- 当前工作上下文记忆
- 关键对象
- 关键关系
- 强写入细节
- 动态增强槽位
- 行业长期偏好槽位
- proposal / diff / rollback
- 人工钉住入口

### 7.3 前台硬规则

- 不把底层 JSON 原样甩给用户
- 不把知识图谱直接退化成一堆 relation 文本
- 不把 sleep 结果写成一段松散大白话
- 不让前台形成第二真相源

---

## 8. 实现边界

这次是“在现有正式链上增强”，不是另起炉灶。

### 8.1 可以新增 / 改造的正式能力

- 图谱到睡眠层的结构化输入整理
- 行业记忆结构增强器
- 动态槽位评估与长期偏好升格 / 降权
- 关键细节强写入服务
- 结构 proposal 真执行器
- `Runtime Center / Knowledge` 的简化 / 结构化读面

### 8.2 明确不做的事

- 不新增向量库作为正式主链依赖
- 不重做知识图谱数据库
- 不把聊天压缩重新升级为共享正式记忆
- 不让模型直接改底层事实
- 不让聊天页承担正式行业记忆结构主入口

---

## 9. 完成标准

这次不能按“多了几个接口 / 多了几个卡片”算完成，必须按闭环算。

### 9.1 功能完成标准

必须同时满足：

1. 图谱真的成为睡眠层输入
2. 通用主槽位 + 动态增强槽位真的生效
3. 长期偏好能升格，也能降权
4. 关键细节强写入真的进入正式主读链
5. `proposal apply` 真正重写 active overlay/profile
6. `Runtime Center` 与 `Knowledge` 能稳定读到新结构

### 9.2 验收标准

必须跑真实长链验收，而不是只跑局部单测：

- 连续多轮同一 `work_context`
- 行业主轴不丢
- 中途插入冲突
- 重启恢复
- 多 agent 并发
- 睡眠整理后次日续跑

### 9.3 终态判断

这轮做完的正确表述应该是：

> 系统已经不是只有“事实 + 图谱 + 睡眠层”三个散件，而是形成了一条可跨行业持续工作的长期记忆组织链。

不正确的表述包括：

- “只是多了几张记忆卡片”
- “只是又加了一套睡眠层摘要”
- “只是图谱多了几个关系”
