# 单行业长期自治载体设计稿

## 1. 目标

本设计用于把 `CoPaw` 从“按行业临时拼团队的聊天执行平台”收口为“单行业长期自治载体”。

核心目标只有一条：

- 一个载体只服务一个行业。
- 一个行业载体只拥有一个主脑。
- 主脑长期负责规划、学习、交流、分配、监督、复盘、进化。
- 主脑永不亲自执行叶子任务。
- 叶子任务只能由长期岗位或临时岗位承担。

行业差异主要体现在知识包、岗位图谱、能力图谱、KPI 和风险边界，不体现在是否拥有自治主链。

## 2. 设计结论

### 2.1 行业载体模型

系统中的自治单位不是“多行业总平台里的一个随手创建团队”，而是：

- `一个行业 = 一个长期载体`
- `一个载体 = 一个行业主脑 + 一组岗位 + 一条长期运行主链`

例如：

- 电商载体：主脑管理客服、运营、美工、库管、研究员
- 股票载体：主脑管理新闻分析员、交易员、风控、研究员

同一套自治骨架适用于不同产业，但每个载体只绑定自己的行业语义，不允许自动漂移成别的行业。

### 2.2 主脑边界

主脑只负责：

- 长期规划
- 聊天交流
- 目标分解
- 分配任务
- 监督执行
- 汇总结果
- 学习与进化

主脑明确不负责：

- 浏览器叶子操作
- 桌面叶子操作
- 文件整理
- 客服回复
- 内容生产与发布
- 下单、交易、删除、转账等外部动作
- 任何直接落地执行

正式规则：

`缺岗 -> 创建 TemporarySeat 或提案新增 CareerSeat；主脑永不亲自下场。`

### 2.3 研究位

每个行业载体默认常驻一个 `ResearchSeat`。

它的职责是：

- 持续观察行业变化
- 跟踪竞品、市场、新闻、政策、热点、平台机制
- 回流给主脑可操作的情报、风险和机会

研究位是主脑的“眼睛”，不是新的战略中心。

## 3. 首次创建流程

### 3.1 用户最小输入

首次创建行业身份时，用户只需要输入四项：

- 行业
- 细分赛道
- 想成为什么
- 一句话目标

例如：

- 行业：电商
- 细分赛道：成人用品
- 想成为什么：电商经营主脑
- 一句话目标：从 0 到稳定出单

### 3.2 系统自动编译内容

系统根据最小输入自动生成以下草案：

- `IndustryBrainIdentity`
- `IndustryKnowledgeProfile`
- `Mission / KPI draft`
- `Industry boundary draft`
- `OperatingLanes`
- `CareerSeats`
- `ResearchSeat`
- `SeatPolicy`
- `BrainPolicy`

用户不需要在第一次创建时手工填写完整行业边界、KPI 体系和组织编制。

### 3.3 初始建队策略

首次自动建队默认采用“平衡型”：

- 1 个主脑
- 1 个常驻研究员
- 该行业的一组核心长期岗位

不走极简空团队，也不走超配团队。

## 4. 身份模型

### 4.1 身份核心层

以下内容属于身份核心，不允许系统自动漂移：

- 行业类型
- 细分赛道主线
- 主脑职业定位

例如电商主脑可以优化为“内容电商一体化经营主脑”，但不能自动漂成股票交易主脑。

### 4.2 身份可优化层

以下内容允许主脑在证据驱动下持续优化：

- 长期使命表述
- KPI 权重
- 行业边界细化
- 岗位分工
- 岗位数量
- 常规工作节奏

### 4.3 跨行业变更

跨行业不属于“优化”，而属于“迁移”。

如果用户要从电商换到股票，应新建新载体，而不是把旧载体直接改种类。

## 5. 岗位模型

### 5.1 CareerSeat

长期岗位，按行业默认编制生成。

特点：

- 可复用
- 可长期持有职责
- 归属某条长期工作面
- 接受主脑分配
- 产出证据与回报

### 5.2 TemporarySeat

临时岗位，用于缺岗补位。

特点：

- 只服务当前任务包
- 任务边界明确
- 有回收条件
- 不持有独立长期使命
- backlog 清空后应退场

### 5.3 Seat 升格规则

当某类临时需求重复出现并形成稳定工作面时，主脑应提出：

- “把临时工升格为长期岗位”

而不是无限继续靠临时工硬撑。

## 6. 运行时主链

### 6.1 正式入口

运行中的人类输入，不等于重新定义主脑身份。

系统要区分两类入口：

- `Identity / Industry Bootstrap`
  - 创建时定义行业主脑身份
- `Operator Runtime Intake`
  - 运行中追加约束、任务、纠偏、确认、加岗、暂停、优先级调整

### 6.2 主脑决策流

运行时，主脑应执行以下流程：

1. 把输入整理为：
   - 目标
   - 约束
   - 风险
   - 证据要求
   - 所需执行面
2. 查询现有长期岗位里是否存在合适 owner
3. 若存在，分派给长期岗位
4. 若不存在，进入缺岗决策
5. 缺岗时绝不允许主脑自己执行
6. 只能三选一：
   - 创建低风险临时工
   - 对高风险临时工创建发起确认
   - 提案新增长期岗位

### 6.3 风险治理

已确认的规则：

- 低风险临时工：可自动创建
- 高风险临时工：需要确认
- 新增长期岗位：需要确认

这与现有 `auto / guarded / confirm` 风险模型保持一致，不新增第四套审批语义。

## 7. 证据与回流

所有执行位都必须向主脑回流：

- 做了什么
- 证据是什么
- 当前阻塞是什么
- 下一步建议是什么

主脑不应只看到字符串总结，必须逐步可见：

- 当前 owner
- 当前风险
- 当前环境
- 当前证据
- 当前结果

## 8. 与现状的关键偏差

当前系统的主要问题不是“行业不够多”，而是边界不稳：

- 主脑与执行位边界不够硬
- 行业实例更像临时团队而非长期载体
- 缺岗时容易错误落到 `execution-core`
- 没有把“缺岗 -> 临时工/长期岗提案”收成正式主链

本设计要求把这些问题改成结构性规则，而不是继续在 prompt 或特判里补洞。

## 9. 非目标

本设计当前不直接承诺：

- 自动完成真实股票交易能力
- 自动完成所有行业的能力安装
- 一次性支持多个行业并行经营

这份设计先解决“单行业长期自治载体”的正确骨架。

## 10. 验收标准

当以下条件成立时，可视为主链设计落地正确：

- 创建身份时只需最小四项输入
- 系统能自动生成行业主脑与核心岗位
- 主脑不会亲自执行叶子任务
- 缺岗时能自动创建低风险临时工
- 高风险缺岗会进入确认流
- 重复临时需求会生成长期岗位提案
- 研究位持续向主脑回流行业观察
- 所有执行结果都能以正式证据回流主脑与前端

## 11. 下一步实现主题

后续实现应优先围绕以下主题展开：

- 主脑身份对象与 seat 对象正式化
- 主脑边界硬化，切断主脑直做叶子活
- 缺岗治理链：现有岗位 / 临时工 / 长期岗提案
- 行业 bootstrap 编译器收口到最小输入模型
- 研究位常驻化与回流主链
- 前端把主脑、岗位、证据、缺岗、提案显式看见
