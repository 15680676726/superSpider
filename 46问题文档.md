## 3. 本轮已修复并核实的问题

### 3.1 `HumanAssistTask` 验收合同已从“伪强语义”修成正式分模式验证

当前状态：`已修复`

本轮已确认：

- `verify_task()` 在直接观察到宿主输入时，会先物化真实 `submitted` 快照，再进入 `verifying`。
- `evidence_verified` 不再允许只靠文本锚点通过，必须有正式 `verification_evidence_refs`。
- `state_change_verified` 不再只靠文本过验，必须满足配置中的状态合同路径。
- `verification_evidence_refs` 会正式写回结果。
- evidence ledger 写入失败不再静默接受；现在会回退到验证前状态，并返回 `verification_record_failed`。

关键落点：

- `src/copaw/state/human_assist_task_service.py`
- `tests/state/test_human_assist_task_service.py`
- `tests/app/test_runtime_human_assist_tasks_api.py`

真实性结论：

- 这一项不再是旧文档里那种“对象模型声明有，真实验收没有”的伪闭环。
- 当前剩余边界主要在 producer/live consumer 的更大链路，不在 `HumanAssistTaskService` 本身的验收合同。

### 3.2 `/runtime-center/industry/{id}` 的 focus truth patching 已收紧到 canonical 边界

关键落点：

真实性结论：

- 当前 focus contract 已明显比上一轮更真实、更收口。

## 4. 部分收口但仍未完全闭环的问题

### 4.1 行业 bootstrap 的一部分 legacy dispatch 已收住，但 kickoff/auto-resume 还没完全切干净

当前状态：`部分修复`

本轮已确认的正向变化：

- public bootstrap 不再把 `auto_activate` 自动膨胀成 legacy goal dispatch。
- chat writeback 新建 schedule 不再扩张 industry instance 的 `schedule_ids` 真相。
- `run_operating_cycle(...)` 已把 task subgraph 传进 cycle/assignment planners。
- 本轮 `bootstrap_lifecycle.py` 已全量通过，因此这里不是“当前回归挂着”的 blocker。

仍然真实存在的问题：

- `service_activation.py`、`service_lifecycle.py` 里仍有大量 `goal_ids / schedule_ids / dispatch_goal_*` 残留路径。
- kickoff / auto-resume 仍没有完全改成 assignment/backlog-native execution。
- `IndustryInstanceRecord` 和 `StrategyMemoryRecord` 仍保留 legacy goal 相关运行事实。

结论：

- 当前是“功能回归已收住，架构债还在”。
- 不能把这一块写成“legacy goal/schedule 主链已经退役完成”。

### 4.2 acquisition decision 的路由回归已修，但 producer/kernel-task 一体化仍未完成

当前状态：`部分修复`

本轮已确认：

- `Runtime Center` approve/reject acquisition decision 现在优先走 canonical dispatcher。
- learning acquisition review-gate approve/reject 现在也优先走 canonical dispatcher。
- acquisition run 现在会把 confirm proposal 正式准入 kernel `waiting-confirm`，后续 approve/reject 会回写真实 kernel result。
- runtime-center actor route 对 sync/async dispatcher 都能兼容；当 kernel task 缺失时，现在会直接返回 `404`，不再偷偷走 acquisition 专属 route fallback。

关键落点：

- `src/copaw/app/routers/runtime_center_routes_actor.py`
- `src/copaw/app/routers/runtime_center_dependencies.py`
- `src/copaw/learning/acquisition_runtime.py`
- `tests/app/test_learning_api.py`
- `tests/app/test_runtime_center_api.py`

仍然真实存在的问题：

- `learning/acquisition_runtime.py` 内部仍保留 compatibility path：当 kernel dispatcher 不可用、或 kernel-backed task/decision 无法建立时，仍可能退回直接构造 `DecisionRequestRecord / TaskRecord`。
- 也就是说，route regressions 与 runtime-center 的长期 fallback 已经收住了，但 producer-side 和 kernel identity 还没有完全并成单一治理链。

结论：

- 当前可以写成“route-level regression 已修复，runtime-center actor route fallback 已删除”。
- 不能写成“DecisionRequest / learning acquisition 已完全统一到单一内核任务模型”。

### 4.3 文档高估口径这轮已纠偏，但不是一次性彻底收口

当前状态：`部分修复`

本轮会同步更新：

## 5. 当前仍然真实存在的高优先级问题

### 5.1 legacy `goal/schedule` 仍是行业主链里的平行事实源

真实问题：

- `IndustryInstanceRecord.goal_ids / schedule_ids`
- `StrategyMemoryRecord.active_goal_ids / active_goal_titles`
- lifecycle / activation 里的 `dispatch_goal_*`

这些事实仍然在表达同类运行状态。

结论：

- 当前不是“无 legacy”，只是“legacy 不再像上一轮那样到处引发直接回归”。

### 5.2 单一真相源依然没有完全建立

真实问题：

- industry instance
- strategy memory
- kernel live runtime
- route projection
- frontend local presentation

仍然有多层同时表达运行态。

结论：

- 当前可以说“对象模型已经比较成型”，不能说“单一真相源已经完成”。

### 5.3 capability execution 仍没有彻底收成单一执行前门

真实问题：

- 读模型收口明显优于执行模型。
- `Runtime Center /external-runtimes/actions` 现在已经改成 `kernel dispatcher submit -> execute_task`，并继承 mount `risk_level`，不再直接绕到 `capability_service.execute_task(...)`。
- 但 `tool / skill / MCP / query-time local delegate / react_agent builtin fallback / browser routine` 仍未完全统一到同一 admission/risk/evidence 前门。

结论：

- 这一块仍然是后续需要继续深挖的硬问题。

### 5.4 Decision / Evidence / Learning 仍不是一条完全统一的治理链

真实问题：

- acquisition 的 runtime-center route fallback 已删，但 learning producer 仍保留 compatibility fallback
- learning / patch / growth 与统一 `EvidenceRecord` 主链还不是完全同构
- 一些治理对象虽然已有正式 record，但 producer/front-door 仍分散

结论：

- 当前能说“比上一轮更一致”，不能说“已经单链治理”。

### 5.5 官方主入口更清晰了，但仍不能认定“唯一前门”

真实问题：

- `/runtime-center/chat/run` 仍然只是最明确的 operator-facing 前门。
- channel、cron、其他 runtime/action 入口仍然存在。

结论：

- 当前更像“主入口更明确”，不是“其他入口全部退役”。

## 6. 重复内容、逻辑不连贯与真实性边界

### 6.1 当前仍可观察到的重复真相源

- `IndustryInstanceRecord.goal_ids / schedule_ids`
- `StrategyMemoryRecord.active_goal_ids / active_goal_titles`
- kernel live runtime 内存态
- 各类 projection/read surface 的派生 focus/environment/owner

### 6.2 当前仍可观察到的逻辑不连贯点

- 行业 public bootstrap 已经不再自动 legacy dispatch，但 kickoff/auto-resume 还保留旧链。
- Runtime Center 的 focus query 已收成 assignment/backlog，但更大范围的 runtime truth 仍未统一到单一 owner。
- `Runtime Center /external-runtimes/actions` 已切回 dispatcher admission，但 query-time delegate / react_agent builtin fallback / system dispatch 仍存在额外绕路面。
- acquisition 的 runtime-center decision route 已经删除专属 fallback，但 learning producer 仍需要为非统一 kernel identity 兜底。

### 6.3 当前对“真实性”的正确表述

可以成立的说法：

- 关键后端主链现在有真实回归结果支撑。
- `bootstrap_lifecycle` 本轮全量通过。
- `HumanAssistTask` 的核心验收合同现在是真实现，不是伪强语义。
- `Runtime Center` 前后端主读面至少在本轮覆盖的 contract 上是连通的。

不能成立的说法：

- “项目已经完全闭环”
- “单一真相源已经完成”
- “所有治理入口已经收成单一主链”
- “legacy goal/schedule 已完全退役”

## 7. 最终定性

一句话定性：

> 当前 CoPaw 已经不是“概念型架构”或“假闭环”；它具备真实可跑、可测的正式主链。但它仍然是一个`关键回归已收住、若干核心债未切净`的部分闭环系统。

更准确的内部表述应该是：

- 正式主链：`能跑`
- 关键回归：`本轮已重新验证`
- HumanAssist 验收合同：`已修复`
- Runtime Center canonical focus query：`已修复`
- acquisition route regression：`已修复，runtime-center route fallback 已删，但 producer 统一未完成`
- external runtime capability front door：`已修复一条真实绕路，但更广泛 capability 前门仍未统一`
- legacy goal/schedule、单一真相源、单一治理链：`仍未完成`

## 8. 下一步优先顺序

建议继续按下面顺序推进：

1. 先切掉 kickoff/auto-resume 对 legacy `goal/schedule` 的主动依赖。
2. 把 acquisition proposal / decision producer 正式并进 kernel task identity，删掉 producer compatibility fallback 的长期合理性。
3. 继续删除 `goal_ids / schedule_ids / active_goal_ids` 这类平行真相源。
4. 继续收紧 capability execution 的统一 admission/risk/evidence 前门，重点清理 query-time delegate、react_agent builtin fallback、system dispatch。
5. 继续做文档真实性治理，把“目标态、代码基线、focused harness、默认回归、live smoke”分层写清。

这才符合当前仓库真正的问题顺序。
