## 1. 审计结论

本次按 10 个独立审计域做了并行排查，并补做了正式测试与源码复核。结论不是“项目没落地”，也不是“已经完全闭环”，而是：

> 当前项目处于“新链中后段真实可跑、前半段和若干执行边界仍混用旧语义、文档与部分读面口径高于真实实现”的`部分闭环`状态。

更具体地说：

- 真链已经存在：
  - `Runtime Center` canonical surface、operator runtime e2e、phase-next autonomy smoke、human-assist API、provider/routine live smoke 都有真实测试通过。
  - `strategy -> lane -> backlog -> cycle -> assignment -> report` 的中后段不是假壳，确实可跑。
  - 环境仓、lease、host recovery、evidence、部分 high-risk confirm chain 也是真实现。
- 但还不能认定“完全闭环”：
  - 行业 bootstrap / control-thread kickoff 仍会主动点燃旧 `goal/schedule` 链。
  - 状态层、运行层、读面层、前端展示层仍有多处双真相源或本地补真相。
  - `HumanAssistTask` 的一部分验收语义是“声明存在、实现未兑现”。
  - 能力系统、风险治理、learning/patch/evidence 还没有收成真正单一主链。

综合评级：

- 闭环程度：`部分闭环`
- 真实性：`中上`
- 文档口径可信度：`中`
- 主链删尾完成度：约 `60%`

## 4. 最高优先级问题

### 问题 1：`HumanAssistTask` 的部分验收语义是假的强语义

严重度：`高`

问题：

- 状态模型已经声明 `anchor_verified / evidence_verified / state_change_verified` 三种验收模式。
- 但 `verify_task()` 实现没有按 `acceptance_mode` 分支，而是统一把文本、evidence refs、payload 扁平化后做 anchor 匹配。
- 验收落库时只写 `verification_payload`，没有把 `verification_evidence_refs` 写回正式结果。
- 这意味着名义上要求“核证据”或“核状态变更”的任务，当前仍可能仅凭聊天文本过验收并恢复执行。

关键证据：

- `src/copaw/state/models_core.py:35`
- `src/copaw/state/models_goals_tasks.py:101`
- `src/copaw/state/models_goals_tasks.py:109`
- `src/copaw/state/human_assist_task_service.py:343-369`
- `src/copaw/state/human_assist_task_service.py:562-573`

影响：

- 宿主接管、人工回执、状态恢复等高价值协作链存在“验收强度低于对象语义”的风险。
- 闭环表面成立，但验收真实性不足。

- 行业激活链仍创建 goals、dispatch goals，并把 `goal_ids/schedule_ids` 写回 industry instance。
- control-thread kickoff 和 auto-resume 也仍直接走 `dispatch_goal_execute_now / dispatch_goal_deferred_background`。

关键证据：

- `src/copaw/industry/service_lifecycle.py:1311-1317`

影响：

- 文档虽已宣称旧 `goal/task/schedule` 主线不再是未来兼容目标，但真实执行入口并未切干净。
- 项目不能认定为“唯一运行主链”。

## 7. 重复内容、重复真相源与逻辑不连贯点

### 7.1 重复真相源

- `IndustryInstanceRecord.goal_ids/schedule_ids`
- `StrategyMemoryRecord.active_goal_ids/active_goal_titles`
- kernel live `_tasks / _terminal_results`
- route layer 注入的 `focus_selection`
- frontend local fallback 的 `currentFocus/currentOwner/currentEnvironment`

这些都在表达同一类正式运行事实，但来源不止一个。

### 7.2 重复口径

- “唯一主链”在 `MASTERPLAN / TASK_STATUS / API_TRANSITION_MAP / V7` 中含义不完全一致。
- Runtime Center canonical read surface 在部分文档中仍混用 `/surface` 与 `/main-brain` 时代语言。
- V7 命名与旧阶段命名之间存在概念叠层，容易让人把“目标态词汇”误读成“当前真实对象”。

### 7.3 逻辑不连贯点

- 文档说旧 goal 主线已退，但行业正式入口仍在发 goal dispatch。
- 文档说 runtime writer 已停用 goal focus，但 runtime sync 仍会写 `current_focus_kind=\"goal\"`。
- 文档说 `industry/service_runtime_views.py` 不再发明 focus，但 read side 仍会在 execution 缺值时补回 `current_focus_*`。
- 文档说 `HumanAssistTask` 恢复链已接回真实消费者，但关键测试仍有 fake consumer。

## 8. 对“真实性”的判断

### 8.1 真实的部分

- 不是 PPT 架构。
- 很多核心对象、state repo、API surface、环境仓、evidence、operator 链都已真实落地。
- 多条 e2e、integration、API、smoke 测试可以证明项目已经超过“概念原型”。

### 8.2 不够真实的部分

- 一些“已验证”叙述实际上只被 contract/fake-harness 锁住。
- 一些“已闭环”叙述依赖读面投影或本地 fallback，不等于写路径已收死。
- 一些“统一对象”只是统一了 record 类型，没有统一 producer / admission / evidence / governance front door。

最终判断：

> 项目整体是真实现，不是假闭环；但“完全闭环、单一真相源、单一执行主链、文档口径与代码现实完全一致”这四件事，目前都还没做到。
